**एसपीए (SPA) के लिए स्वचालित डिज़ाइन निदान:** एआई-यूआई आपके चल रहे एप्लिकेशन का विश्लेषण करता है, आपके दस्तावेज़ों को पढ़ता है, और आपको सटीक रूप से बताता है कि किन प्रलेखित सुविधाओं में कोई खोज योग्य यूआई (यूजर इंटरफेस) प्रवेश बिंदु नहीं है—और किन यूआई तत्वों का बिल्कुल भी दस्तावेज़ नहीं है।

यह अनुमान नहीं लगाता। यह वास्तविक ब्राउज़र इंटरैक्शन से एक "ट्रिगर ग्राफ" बनाता है, विशेषताओं को ट्रिगर्स से निश्चित रूप से जोड़ता है, और एक "डिजाइन-मैप" तैयार करता है जिसमें कार्रवाई योग्य निष्कर्ष होते हैं: "दिखाएं," "कम महत्व दें," "यथावत रखें," "एकत्र करें।" फिर, यह सुधार की पुष्टि करता है।

## यह क्या करता है।

```
README says "ambient soundscapes"  →  atlas extracts the feature
Probe clicks every button           →  "Audio Settings" trigger found
Diff matches feature to trigger     →  coverage: 64%
Design-map says: must-surface 0     →  all documented features are discoverable
```

एआई-यूआई, दस्तावेज़ों में किए गए वादों और वास्तविक यूजर इंटरफेस के बीच के अंतर को कम करता है।

## स्थापित करें।

```bash
git clone https://github.com/mcp-tool-shop-org/ai-ui.git
cd ai-ui
npm install
```

इसके लिए नोड.जेएस 20 या उससे ऊपर का संस्करण और एक चल रहा डेवलपमेंट सर्वर, "प्रोब" और "रनटाइम-इफेक्ट्स" कमांडों के लिए आवश्यक है।

## शुरुआत कैसे करें।

```bash
# 1. Parse your docs into a feature catalog
ai-ui atlas

# 2. Crawl your running app
ai-ui probe

# 3. Match features to triggers
ai-ui diff

# Or run all three in sequence:
ai-ui stage0
```

आउटपुट फ़ाइलें `ai-ui-output/` नामक फ़ोल्डर में जमा होती हैं। "डिफ" रिपोर्ट आपको बताती है कि क्या मेल खाता है, क्या गायब है, और क्या प्रलेखित नहीं है।

## आदेश।

| आदेश। | यह क्या करता है। |
| ज़रूर, मैं आपकी मदद कर सकता हूँ। कृपया वह अंग्रेजी पाठ प्रदान करें जिसका आप हिंदी में अनुवाद करवाना चाहते हैं। | कृपया वह अंग्रेजी पाठ प्रदान करें जिसका आप हिंदी में अनुवाद करवाना चाहते हैं। मैं उसका सटीक और उचित अनुवाद करने के लिए तैयार हूं। |
| `atlas` | दस्तावेज़ों (जैसे README, CHANGELOG, आदि) का विश्लेषण करके एक सुविधा सूची (फीचर कैटलॉग) तैयार करें। |
| `probe` | उपलब्ध यूजर इंटरफेस (यूआई) को स्कैन करें और प्रत्येक इंटरैक्टिव तत्व को रिकॉर्ड करें। |
| `surfaces` | वेबस्केच कैप्चर से सतहों को निकालें। |
| `diff` | मैच एटलस की विशेषताओं की तुलना जांच ट्रिगर्स से करें। |
| `graph` | "प्रोब, सतहों और अंतर डेटा का उपयोग करके ट्रिगर ग्राफ बनाएं।" |
| `design-map` | सतह की वस्तुओं की सूची तैयार करें, विशेषताओं का मानचित्र बनाएं, कार्य प्रक्रियाओं का विवरण दें, और सूचना वास्तुकला (सूचना संरचना) का प्रस्ताव प्रस्तुत करें। |
| `compose` | "डिफ" और ग्राफ डेटा का उपयोग करके, एक "सरफेसिंग प्लान" (सतह बनाने की योजना) तैयार करें। |
| `verify` | जांच करें कि पाइपलाइन के दौरान उत्पन्न होने वाली चीज़ें (आर्टिफैक्ट्स) सही हैं या नहीं - निरंतर एकीकरण (सीआई) के लिए पास/फेल का निर्धारण करें। |
| `baseline` | सत्यापन के आधारभूत मापदंडों को सहेजें/तुलना करें। |
| `pr-comment` | "आर्टिफैक्ट्स (artifacts) से एक ऐसा मार्कडाउन कमेंट तैयार करें जो पीआर (PR - Pull Request) के लिए उपयुक्त हो।" |
| `runtime-effects` | "क्लिक" पर प्रतिक्रिया वास्तविक ब्राउज़र में होती है, और देखे गए दुष्प्रभावों को रिकॉर्ड किया जाता है। |
| `runtime-coverage` | प्रत्येक ट्रिगर के लिए कवरेज मैट्रिक्स (जांच की गई/ उजागर की गई/ देखी गई)। |
| `replay-pack` | सभी कलाकृतियों को एक साथ मिलाकर एक ऐसा पैकेज बनाएं जिसे दोबारा इस्तेमाल किया जा सके और जिसके परिणाम दोहराए जा सकें। |
| `replay-diff` | दो "रिप्ले पैक" की तुलना करें - दिखाएं कि क्या बदला है और क्यों। |
| `stage0` | "एटलस, प्रोब और डिफ को क्रम से चलाएं।" |
| `init-memory` | निर्णय लेने की प्रक्रिया को ट्रैक करने के लिए, खाली मेमोरी फ़ाइलें बनाएं। |

## कॉन्फ़िगरेशन।

अपने प्रोजेक्ट के मुख्य फ़ोल्डर में `ai-ui.config.json` फ़ाइल बनाएं:

```json
{
  "docs": { "globs": ["README.md", "CHANGELOG.md", "docs/*.md"] },
  "probe": {
    "baseUrl": "http://localhost:5173",
    "routes": ["/", "/settings", "/dashboard"]
  },
  "featureAliases": {
    "dark-mode-support": ["Theme", "Dark mode"]
  },
  "goalRules": [
    { "id": "settings_open", "label": "Open Settings", "kind": "domEffect", "dom": { "textRegex": "Settings" }, "score": 2 }
  ]
}
```

सभी फ़ील्ड वैकल्पिक हैं - उचित डिफ़ॉल्ट मान लागू किए गए हैं। पूर्ण संरचना (schema) के लिए `cli/src/config.mjs` फ़ाइल देखें।

### लक्ष्य निर्धारण के नियम।

उन एसपीए (SPA - सिंगल पेज एप्लिकेशन) के लिए जहां यूआरएल (URL) नहीं बदलते हैं, रूट-आधारित लक्ष्य (लक्ष्य नियम) बेकार होते हैं। लक्ष्य नियमों के माध्यम से, आप सफलता को "अवलोकनीय प्रभाव" के रूप में परिभाषित कर सकते हैं:

| Kind | माचिसें। | उदाहरण। |
| "The company is committed to providing high-quality products and services."

अनुवाद:

"कंपनी उच्च गुणवत्ता वाले उत्पाद और सेवाएं प्रदान करने के लिए प्रतिबद्ध है।" | "The company is committed to providing high-quality products and services."

अनुवाद:

"कंपनी उच्च गुणवत्ता वाले उत्पाद और सेवाएं प्रदान करने के लिए प्रतिबद्ध है।" | "The company is committed to providing high-quality products and services."

अनुवाद:

"कंपनी उच्च गुणवत्ता वाले उत्पाद और सेवाएं प्रदान करने के लिए प्रतिबद्ध है।" |
| `storageWrite` | लोकल स्टोरेज/सेशन स्टोरेज में डेटा लिखने की प्रक्रिया। | `{ "keyRegex": "^user\\.prefs\\." }` |
| `fetch` | विधि/यूआरएल/स्थिति के अनुसार एचटीटीपी अनुरोधों की जानकारी। | `{ "method": ["POST"], "urlRegex": "/api/save" }` |
| `domEffect` | डीओएम में होने वाले बदलाव (जैसे कि पॉप-अप विंडो खोलना, सूचना संदेश प्रदर्शित करना, आदि)। | `{ "textRegex": "saved" }` |
| `composite` | और विभिन्न प्रकार के। | "सेटिंग्स सहेजी गईं" के लिए स्टोरेज और डोम। |

नियमों के अनुसार, लक्ष्य प्राप्ति के लिए रनटाइम डेटा (रनटाइम प्रभाव + ग्राफ) का होना आवश्यक है। यदि कोई डेटा उपलब्ध नहीं है, तो लक्ष्यों का मूल्यांकन नहीं किया जा सकता है—जिससे कोई गलत परिणाम नहीं आएगा।

## डिजाइन-मैप का आउटपुट।

`design-map` कमांड चार प्रकार की आउटपुट फ़ाइलें (या उत्पाद) उत्पन्न करता है:

- **सतही तत्वों की सूची:** प्रत्येक इंटरैक्टिव तत्व को उसके स्थान के अनुसार वर्गीकृत किया गया है (मुख्य नेविगेशन, सेटिंग्स, टूलबार, इनलाइन)।
- **विशेषताओं का मानचित्र:** प्रत्येक प्रलेखित विशेषता की खोज क्षमता, प्रवेश बिंदु और अनुशंसित कार्रवाई का विवरण।
- **कार्य प्रक्रियाएं:** अनुमानित नेविगेशन श्रृंखलाएं, जिनमें लूप का पता लगाना और लक्ष्य ट्रैकिंग शामिल है।
- **सूचना वास्तुकला प्रस्ताव:** मुख्य नेविगेशन, द्वितीयक नेविगेशन, अनिवार्य रूप से प्रदर्शित होने वाले तत्व, प्रलेखित लेकिन प्रदर्शित न किए जाने वाले तत्व, और रूपांतरण पथ।

| `merge` | विभिन्न मार्गों में डुप्लिकेट फ़ीचर नाम हैं - इन्हें एक साथ समेकित करें। |
| `skip` | यह वास्तव में कोई फ़ीचर नहीं है (वाक्य जैसा नाम, जिसका कोई ठोस आधार नहीं है)। |

## पाइपलाइन।

पूरी पाइपलाइन अनुक्रम:

```
atlas → probe → diff → graph → design-map
                 ↓
          runtime-effects → graph --with-runtime → design-map (with goals)
                                                        ↓
                                                   replay-pack → replay-diff
```

प्रत्येक चरण पिछले चरण के आउटपुट को `ai-ui-output/` से पढ़ता है। पाइपलाइन नियतात्मक है - समान इनपुट समान आउटपुट उत्पन्न करते हैं।

## सीआई (CI) एकीकरण।

```bash
# Run pipeline + verify in CI
ai-ui stage0
ai-ui graph
ai-ui verify --strict --gate minimum --min-coverage 60

# Exit code 0 = pass, 1 = user error, 2 = runtime error
```

मशीन-पठनीय आउटपुट के लिए `--json` का उपयोग करें। थ्रेशोल्ड को सेट करने के लिए `baseline --write` का उपयोग करें।

## खतरे का मॉडल।

AI-UI आपके डेवलपमेंट सर्वर पर स्थानीय रूप से चलता है। यह निम्नलिखित नहीं करता है:
- बाहरी सेवाओं को डेटा भेजना।
- आपके स्रोत कोड या कॉन्फ़िगरेशन को संशोधित करना।
- कॉन्फ़िगर किए गए `baseUrl` और दस्तावेज़ फ़ाइलों के बाहर किसी भी चीज़ तक पहुंचना।
- नेटवर्क एक्सेस की आवश्यकता होती है (सभी विश्लेषण स्थानीय रूप से किए जाते हैं)।

`runtime-effects` कमांड प्लेराइट ब्राउज़र में वास्तविक बटनों पर क्लिक करता है। यह सुरक्षा नियमों का पालन करता है:
- मेल खाने वाले 'निषेध' पैटर्न (हटाना, हटाना, नष्ट करना, आदि) को छोड़ दिया जाता है।
- `data-aiui-safe` विशेषता ज्ञात रूप से सुरक्षित ट्रिगर्स के लिए सुरक्षा को ओवरराइड कर सकती है।
- `--dry-run` मोड में, क्लिक करने के बजाय, केवल होवर किया जाता है।
